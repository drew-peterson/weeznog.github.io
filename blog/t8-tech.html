Test Driven Development

What is Testing in Ruby?

Testing in ruby or programming is just like any other form of testing, check to see if you do X you get the expected results Y. We write ruby code to acheive a purpose to complete a goal. Lets say we wanted to make a programe that takes any 2 numbers and adds them together: def add (a,b)
     a + b
end

p add(1,2) We would expect that running the above code would print to the screen the number 3. So p add(1,2) was a mini test, with numbers 1 and 2 the results should be 3. But what if you needed to add 2 different numbers together 100 times. when all the results are printed to the screen, can you keep track of what was added and what the proper result would be? I certainly could not...

if you are glancing at a list of numbers such as 1,234,56,7675,234,1,2,3. Think of these as the results of adding 2 numbers together several times and these are the results. Are these the correct answers, well you could go through the code and look but wouldnt this: true true true true true false true true be a better return alternative. So we immeditaly see that the 6th example failed and then we could look at the 6th call of the function p add(x,y)

The above is kinda hard to picture possibly without writting and printing to the screen a bunch of times. But the take away should be, when testing code it should return true or false.

writting tests that return true or false

using the code above as the function we want to run lets create a test that will return true or false. puts "test 1: "
puts add(2,2) == 4
puts add(1,2) == 4

###### Console #####

false We can use the equality symbol == to return a true..false value.

we can of course set up a more automated test that uses random numbers and checks for specific numbers using built in math, however for the basic understanding of ruby tests this is a good place to start.

Ok then what is test driven development

Test driven development is the idea of writing tests first that return true or false based on what you think your code should return before you even write the code. by writting a serious of tests before you even write 1 line of code, you can check to see how your code is progressing as you write it. You type out a few lines.... run the test is returns false, you know your code is not working up to snuff yet.

lets do another example: let says we want to now create a function that substracts 2 numbers and returns the results. If we want to apply test driven development, we should start by writting somes tests. subtraction test 1:
puts subtraction(6,2) == 4 #this should be true
puts subtraction(6,2) == 3 #this should be false because we know basic math we know that 3 - 1 is 2 and 3-1 is not 5. With these tests in mind we can know start writting the function. def subtraction(num1, num2)
    num1 / num2
end ok now lets run it... subtraction test 1:
false
true Ok why did the first one fail and the 2nd one return true??? well if we look at the code we would see that we made a mistake. Instead of num1 - num2 we did num1 / num2 so 6 / 2 would not return 4 however 6 / 2 does return 3 so that is why the first test faild and the secound one did not.

Why do we use test drive development? to catch mistakes that we make in our code, before its too late. if we made the above code more complex and added some functions that rely on the answer returned from subtracation, how would we ever narrow down the error quickly? If you start with a test, and run often, as long as your tests are correct. You can code with cofidence knowing that you can quickly check to see if you are doing the correct thing.
